database

概述

数据库的概念

  • 数据库 database

存储和管理一组数据，通常按照某种数据模型来组织

  • 数据库系统 database management system

管理数据库的系统

  • 关系模型 relational model

关系模型中的实体和联系都可以用“表”来表示

  • 关系数据库 relational database

采用关系模型的数据库

  • 关系数据库系统 relational database management system

支持关系模型为核心模型的数据库系统

表（table/relation）

  • table/relation
  • 列（column）：一个属性，有明确的数据类型
  • 行（row）：一个记录（tuple，record）

表的数学定义

K列的表：{<t1,…,tk>|t1 belong D1,…,tk belong Dk}

Schema vs. Instanse

  • Schema:模式/类型

只需要定义一次，例如：
Student(ID integer,Name varchar(20),GPA float)

  • Instance: 实例/具体取值

具体存储哪些记录，每个列的具体值

关系模型

  • 优点：数据独立性

数据组织和存储的改变对于应用是隔离的

ER模型（Entity Relationship model）实体联系模型

实体

  • 实体（Entity）
    现实世界中的对象，可以与其他对象区分

  • 实体集（Entity Set）
    相似实体形成的集合

  • 实体的属性（Attribute）
    一个实体集的每个实体都有的属性

erDiagram

    EntitySet{
        int attribute1 PK
        float attribute2
    }
  • 弱实体
    菱形加粗，实体框加粗，部分键下划线用虚线

实体集和属性的表示：

  • 键
    某些属性是特殊的，需要特别标记

联系

如果联系是非堆成的，可以在边上标注联系中的角色。

实体集与联系集的关系

  • 实体参与一个联系，如果仅能参与一次，则需要加入一个箭头，即键约束（key constraint）
flowchart LR

    E1[EntitySet1]
    E2[EntitySet2]

    R{relationship}

    E1 ---> R
    E2 --- R

    A((attribute))
    R --- A
  • 如果可以多个联系，则不用箭头

粗线：每个实体都有联系

类层次

用一个倒三角（IS A）表示实体集之间的继承关系

类层次的约束

  • 交迭约束（overlap）
    两个子类之间是否可以有交集？存在共同的实体？
    默认：不相交（交集为空）
    如有，需要文字特殊说明

  • 覆盖约束（covering）
    多个子类的并集是否覆盖了父实体集？
    默认：没有覆盖约束（并集不是全集）
    如有，需要文字特殊说明

建模设计原则

  • 实体 vs. 属性
    考虑两方面，只有一个值还是多个值
    是简单的值，还是丰富的内部结构
