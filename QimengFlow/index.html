gen_ram.py分析

不参与仿真运行本身，生成一个 .v 模块，默认输出ram_genvar_style.v,是一个参数化 Verilog RAM 生成器。

  • 核心功能
    生成多读口、多写口、分bank的RAM
    RAM 存储体用 reg [DATA-1:0] bank_x [0:DEPTH-1] 实现
    写入是时序逻辑：always @(posedge clk)。
    普通读是组合逻辑：always @(*)。
    支持同拍写读 bypass：读地址命中本拍写端口时，读数据直接返回写数据，而不是旧 RAM 内容。
    支持 full-table read：把整个 RAM 表展平成一个大向量输出。
  • 地址拆分

  • prf和ram的交互

  • PRF -> RAM 读请求
    PRF 发出 src1_preg/src2_preg 地址，请 RAM 读对应物理寄存器值。

  • RAM -> PRF 读数据返回
    RAM 根据 reg_file_addr_0/1 读出数据，再送回 PRF。
  • PRF -> RAM 写请求
  • ram/prf连接关系

下面这些不是 RAM 存储本体信号，但会影响 PRF 最终读出的值：

它们判断当前读地址是否等于写回/旁路目的寄存器。如果命中，read_operand_with_bypass() 会优先用写回或 EXU bypass 数据，而不是 RAM 读出的旧数据。

aag_to_cpp

目录：/BSD/src/aag_to_cpp

  • 根据README跑一遍.v->aag->cpp流程

    如果报错的说path里没有yosys的话，需要下载并将其置于path

  • 直接用top跑（filename=”top_function_layer”）

“看起来卡住”的主要原因：resolve_dependencies() 对 1628 个文件建立模块表之后，在队列里每处理一个模块，都把所有模块名拿去对当前文件做一次正则搜索。依赖一多就是百万级正
则扫描，而且没有进度输出。

  • 解决方法

    1. 每个 function_layer_0_nodes_x_y.v 单独作为 top 生成 C++。
    2. 每个小模块只负责自己的输出范围，比如 o_index[23:15]。
    3. 最后自动生成一个 top_function_layer 的 C++ wrapper，把同一个输入 i/pi 传给各个小模块，再把各自输出拼到总输出数组对应区间。
    4. io_generator_outer_single(bit_index) 也可以根据 bit_index 落在哪个区间，只调用对应的小模块，而不是全量跑。

即新增一个 zstart_div.py 负责解析 top、跑每个子模块、重命名函数避免冲突、生成总 wrapper；
zstart_div.sh 作为入口包装它
建议先跑一个小测试：
sh zstart_div.sh top_function_layer –limit 1 –no-vec
如果这个能完成，会生成：
output_cpp/div_modules/function_layer_0_nodes_0_0.h
output_cpp/top_function_layer_div.h
确认没问题后，再跑全量：
sh zstart_div.sh top_function_layer –no-vec
如果中途断了，可以续跑：
sh zstart_div.sh top_function_layer –no-vec –skip-existing
